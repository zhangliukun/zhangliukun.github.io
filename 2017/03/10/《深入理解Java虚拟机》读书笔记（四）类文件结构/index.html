概述

如今编写成二进制本地机器码已经不是唯一的选择了，越来越多的程序语言选择了与操作系统和机器指令集无关的、平台中立的格式作为程序编译以后的存储格式。各种不同的平台的虚拟机与所有的平台统一使用的程序存储格式是“字节码（ByteCode）”，实现语言无关性的基础仍然是虚拟机和字节码存储格式。Java 虚拟机不和任何语言绑定，它只和“Class 文件”这种特定的二进制文件格式所关联，Class 文件中包含了 Java 虚拟机指令集和符号表以及若干其他的辅助信息。主要能编译成 Class 文件，Java 虚拟机并不关心来源是何种语言，所以Glojure、Groovy、Scala、JRuby、Jython 这些能够编译成 Class 文件的语言都可以在 Java 虚拟机上面运行。

Class 类文件的结构

Class 文件是一个以 8 位字节为基础单位的二进制流，各个数据紧凑排序没有分隔符，当需要占用 8 位字节以上空间的数据项时会按照高位在前的方式分割成若干个 8 位字节进行存储。
Class 文件格式采用的是类似 C 语言结构体的伪结构来存储数据，这种伪结构只有两种数据类型：

  • 无符号数：属于基本的数据类型，以u1、u2、u4、u8 来分别代表1、2、4、8 个字节的无符号数，无符号数可以用来描述数字、索引引用、数量值或者按照 UTF-8 编码构成的字符串值。
  • 表：表示由多个无符号数或者其他表作为数据项构成的复合数据类型，所有表都习惯性的以“_info”结尾，用来描述有层次关系的复合结构的数据。整个 Class 文件本质上面就是一张表。
类型 名称 数量

u2 major_version 1

无论是无符号数还是表，当需要描述同一类型但是数量不定的多个数据的时候，经常会使用一个前置的容量计数器加上若干个连续的数据项的形式，这时称这一系列连续的某一个类型的数据为某一个类型的集合。由于 Class 文件结构是被严格限定的，所以哪个字节代表什么含义，长度是多少，顺序如何都不能被改变，接下来分析一下表中的各个数据项的含义。

魔数与 Class 文件的版本

每个 Class 文件的头 4 个字节被称为魔数（Magic Number），它的唯一作用是确定这个文件是否是一个能够被虚拟机接受的 Class 文件。很多的文件存储标准中都用魔数来进行身份识别，比如在图片格式 gif 或者 jpeg 文件头中都存在魔数。使用魔数而不是扩展名来区分文件的主要原因是基于安全的考虑，毕竟文件后缀可以人为的所以改动。Class 文件的魔数值为 0xCAFEBABE，紧随着魔数的4个字节为 Class 文件的版本号，Java 的版本号是从45开始的，高版本的 JDK 能够向下兼容以前的版本的 Class 文件，但不能兼容以后版本的 Class 文件。

通过 Hex Friend 工具查看这个 class 文件的分析，我们可以看到这个文件开头四个字节的十六进制表示的是 0xCAFEBABE，第5和第6个字节表示的次版本号的值是 0x0000，主版本号的值为 0x0033，即十进制的51，跟我使用的 JDK 1.7 的版本符合。

常量池

紧接着主次版本号之后的就是常量池入口，常量池可以理解为 Class 文件之中的资源仓库，它是 Class 文件结构中与其他项目关联最多的数据类型，也是占用 Class 文件空间最大的数据项目之一，同时它是在 Class 文件中第一个出现的表类型数据项目。
由于常量池中常量的数量是不固定的，所以在常量池入口需要放置一项 u2 类型的数据，代表常量池数值（constant_pool_count），但这个容量计数是从1开始计数的，还是看图2中的对于常量池的数值的十六进制 0x0029，它是十进制的41，这就代表常量池中有40个常量，索引值的范围为1~41。在 Class 文件格式规范制定的时候，设计者将第0项常量空出来来满足后面某些指向常量池的索引值的数据在特定情况下需要表达“不引用任何一个常量池项目”的含义，这种情况下就可以把索引值置为0来表示。Class 文件结构中只有常量池的容量计数是从1开始的。
常量池主要存放两大类常量：字面量（Literal）和符号引用（Symbolic References）。字面量比较接近于 Java 语言层面的常量概念，如文本字符串、声明为 final 的常量值等，符号引用则属于编译原理方面的概念，包括下面三类常量：

  • 类和接口的全限定名（Fully Qualified Name）
  • 字段的名称和描述符（Descriptor）
  • 方法的名称和描述符
    Java 代码在进行 Javac 编译的时候不像 C、C++ 有“连接”这个步骤，而是在虚拟机加载 Class 文件的时候进行动态连接。因此，在 Class 文件中不会保存各个字段的最终内存布局信息，因此这些字段、方法的符号引用不经过运行期转换的话无法得到真正的内存入口地址，也就无法直接被虚拟机使用。当虚拟机运行的时候，需要先从常量池获得对应符号的引用，再在类创建时或者运行时解析、翻译到具体的内存地址中。
    常量池中每一项常量都是一个表，到 JDK 1.7时，共有14种结构各不相同的表结构数据，这些表都有一个共同的特点，就是表开始的第一位是一个 u1 类型的标识位（tag），代表当前这个常量属于哪种常量类型。具体含义如下表所示：

另外，由于 Class 文件中方法、字段都需要引用 CONSTANT_Utf8_info 类型来描述名称，这的 u2 类型的 length 就是所能表达就是最长的值为 65535，所以 Java 程序中如果定义了超过 64KB 英文字符的变量或者方法名，将不能编译。

访问标志

在常量池之后，是两个字节代表的访问标志（access_flags），这个标志用于识别一些类或者接口层次的访问信息，具体的含义表如下所示：

标志名称 标志值 含义
ACC_PUBLIC 0x0001 是否是public
ACC_FINAL 0x0010 是否被声明为final，只有类可以设置
ACC_SUPER 0x0020 是否允许使用invokespecial字节码指令的新语义，JDK1.0.2之后编译出来的类的这个标志默认为真
ACC_INTERFACE 0x0200 标识是一个接口
ACC_ABSTRACT 0x0400 是否是abstract，对于接口和抽象类来说为真，其他类都为假
ACC_SYNITHETIC 0x1000 标识这个类并非由用户代码产生
ACC_ANNOTATION 0x2000 标识这是一个注解
ACC_ENUM 0x4000 标识这是一个枚举

如何计算具体的访问标志的值呢？假设一个类被声明为 ACC_PUBLIC 和 ACC_FINAL，那么这两个标志应该为真，最后的值为 0x0001|0x0021 = 0x0021 ，最后的访问标志就是 0x0021。

类索引、父类索引与接口索引集合

类索引（this_class）和父类索引（super_class）都是一个 u2 类型的数据，而接口索引集合（interfaces）是一组 u2 类型的数据的集合，Class 文件中由这三项数据来确定这个类的继承关系。类索引用来确定这个类的全限定名，父类索引用来确定弗雷的全限定名。除了 java.lang.Object 之外，父类索引只有一个，且父类索引都不为0。接口索引集合就是用来描述这个类实现了哪些接口。
类索引和父类索引引用两个 u2 类型的索引值表示，它们各自指向一个类型为 CONSTANT_Class_info 的类描述符常量，通过 CONSTANT_Class_info 类型常量中的索引值可以找到定义在 CONSTANT_Utf8_info 类型的常量中的全限定名字符串。
对于接口索引集合，入口的第一项–u2 类型的数据为接口计数器（interfaces_count）表示索引表的容量。如果没有实现接口则计数器值为0，后面的接口索引表不占用任何字节。

字段表集合

字段表（field_info）用于描述接口或者类中声明的变量。字段包括类级变量以及实例级变量，但不包括局部变量。描述一个字段可以包含很多信息：字段的作用域、实例变量还是类变量、可读性、并发可见性、是否可序列化、数据类型，上面的信息中，各个修饰符都是布尔值，很适合使用标识位来表示。而字段叫什么名字、字段被定义为什么数据类型，这些都无法固定，只能引用常量池中的常量来描述。

类型 名称 数量
u2 access_flags 1
u2 name_index 1
u2 descriptor 1
u2 attributes_count 1
attribute_info attributes attributes_count

字段表修饰符放在 access_flags 项目中，它与类的 access_flags 项目是非常类似的，都是一个 u2 的数据类型。就不再贴出表结构了。跟随 access_flag 标志的是两项索引值：name_index 和 descriptor_index，它们都是对常量池的引用，分别代表着字段的简单名称以及字段和方法的描述符，下面描述以下这三个特殊的字符串的概念：
全限定名和简单名称：这个很好理解， 上述代码中“com/company/Main”就是这个类的全限定名，只不过将“.”换成了“/”。简单名称是指没有类型和参数修饰的方法或者字段名称，比如上述代码中的 getName 就是 getName() 方法的简单名称。
根据描述符的规则，基本数据类型和无返回值的 void 类型都用一个大写的字符来表示，而对象类型则用字符 L 加对象的全限定名来表示，如下表：

标识字符 含义 标识字符 含义
B byte J long
C char S short
D double Z boolean
F float V 特殊类型void
I int L 对象类型，如 Ljava/lang/Object

对于数组类型，每一个维度都使用一个前置的”[“字符来描述，比如一个“java.lang.String[][]“类的二维数组，将被记录为：”[[Ljava/lang/String”，一个整型的数组“int[]“将被记录为“[I“。
用描述符来描述方法的时候，按照县参数列表，后返回值的顺序进行描述，参数列表按照参数的严格顺序放在一组小括号“（）”内，比如方法 int indexOf(char[] source)的描述符为“([C)I”。

方法表集合

Class 文件存储格式对方法的描述和对字段的描述几乎完全一致，方法表的结构同字段表一样，依次包括了访问标志（access_flag）、名称索引（name_index）、描述符索引（descriptor_index）、属性表集合（attributes）几项。

类型 名称 数量
u2 access_flags 1
u2 name_index 1
u2 descriptor 1
u2 attributes_count 1
attribute_info attributes attributes_count

可以看到和字段表一样，只是在访问标志和属性集合的值的可选项中优点区别，因为 volatile 关键字和 transient 关键字不能修饰方法，所以方法表中的访问标志中没有了 ACC_VOLATILE 标志和 ACC_TRANSENT 标志。方法独有的修饰符同理。

其中，方法中的代码经过编译器编译成字节码指令以后存放在方法属性表集合中一个名为“Code”的属性里面。与字段表集合相对应得，如果父类方法在子类中没有被重写，方法表集合中就不会出现来自父类的方法信息，但同样的，有可能会出现由编译器自动添加的方法，最典型的就是类构造器““方法和实例构造器”“方法。

属性表集合

属性表（attribute_info）出现过许多次了，在 Class 文件、字段表、方法表都可以携带自己的属性表集合。与 Class 文件中其他的数据项目要求严格的顺序、长度和内容不同，属性表相对来说限制宽松一些，只要不与现有属性名重复，任何人实现的编译器都可以向属性表中写入自己定义的属性信息，Java 虚拟机运行时会忽略掉它不认识的属性，为了能正确解析 Class 文件，预定义了虚拟机实现应当能够识别的属性，在 Java 虚拟机规范 7 中，预定义的属性有 21 项，现在只列出最常用的9项下表所示。

属性名称 使用位置 含义
Code 方法表 Java 代码编译成的字节码指令
ConstantValue 字段表 final 关键字定义的常量值
Deprecated 类、方法表、字段表 被声明为 deprecated 的方法和字段
Exceptions 方法表 方法抛出的异常
EnclosingMethod 类文件 仅当一个类为局部类或者匿名类时才能拥有这个属性，这个属性用于标识这个类所在的外围方法
InnerClass 类文件 内部类列表
LineNumberTable Code 属性 Java 源码的行号与字节码指令的对应关系
LocalVariableTable Code 属性 方法的局部变量描述
SourceFile 类文件 记录源文件名称
Synthetic 类、方法表、字段表 标识方法或者字段为编译器自动生成的

对于每个属性，它的名称需要从常量池中引用一个 CONSTNT_Utf8_info 类型的常量来表示，而属性表的结构则是完全自定义的，只需要通过一个 u4 的长度属性去说明属性所占用的位数即可，一个符合规则的属性表应该满足如下的结构：

类型 名称 数量
u2 attribute_name_index 1
u4 attribute_length 1
u1 info attribute_length

Code 属性

Code 属性出现在方法表的属性集合之中，但是并非所有的方法表都存在这个属性，如接口和抽象类中的方法不存在 Code 属性，Code 属性如下所示：

类型 名称 数量

u4 attribute_length 1
u2 max_stack 1
u2 max_locals 1
u4 code_length 1
u1 code code_length
u2 exception_table_length 1
exception_info exception_table exception_table_length
u2 attributes_count 1
attribute_info attributes attributes_count
  • atttibute_name_index是指向了 CONSTANT_Utf8_info 型常量的索引，常量值固定为 “Code”，它代表了该属性的名称，attribute_length 指示了该属性的全部长度，由于属性索引和属性长度占了6个字节，所以后面所有的属性值的长度就为整个属性表的长度减去 6 个字节。
  • max_stack 代表了操作数栈的最大深度，虚拟机运行时根据这个来分配栈帧中的操作数深度。
  • max_locals 代表了局部变量表所需的存储空间，单位为 Slot，Slot 是虚拟机为局部变量表分配所使用的最小单位。
  • code_length 和 code 用来存储 Java 源程序编译后生成的字节码指令。code_length 代表字节码的长度，code 是存储字节码指令的字节流，每个指令是u1 类型的单字节。虽然 code_length 是一个 u4 类型的长度值，但是虚拟机中规定一个方法不允许超过65535条字节码指令，所以实际上只有 u2 的长度，这个长度中包含了所有指令。

Exceptions 属性

这里的 Exception 属性是在方法表中与 Code 属性平级的一项属性，Exception 属性的作用是列举处方法表中可能抛出的受检查异常，也就是方法描述时在 throws 关键字后面列举的异常，结构表如下所示：

类型 名称 数量
u2 attribute_name_index 1
u4 attribute_length 1
u2 number_of_exceptions 1
u2 exception_index_table number_of_exceptions

number_of_exceptions 表示可能抛出的异常的种类，每一个受检查异常使用一个 exception_index_table 表示，这是一个指向常量池中 CONSTANT_Class_info 型常量的索引，代表了类型。

LineNumberTable 属性

这个属性用来描述 Java 源码行号与字节码行号之间的对应关系，并不是运行时必须的属性，但默认会生成到 Class 文件之中，如果选择不生成这些信息最主要的影响就是程序抛出异常时堆栈中不会显示出错的行号，也无法在调试时按照源码来设置断点。结构表如下：

类型 名称 数量
u2 attribute_name_index 1
u4 attribute_length 1
u2 line_number_table_length 1
line_number_info line_number_table line_number_table_length

其中line_number_table 是一个数量为 line_number_table_length、类型为 line_number_info 的集合，line_number_info 表包括了 start_pc 和 line_number 两个 u2 类型的数据项，前者是字节码行号，后者是 Java 源码行号。

LocalVariableTable 属性

这个属性用于描述栈中局部变量表中的变量与 Java 源码中定义的变量之间的关系，也不是运行时必须的属性，默认也会生成到 Class 文件汇总，如果没有生成这项信息主要的影响就是当其他人引用这个方法的时候，所有的参数名称都会丢失，IDE 会使用占位符来替代原有的参数名，调试期间无法根据参数名称从上下文中获得参数值。

类型 名称 数量
u2 attribute_name_index 1
u4 attribute_length 1
u2 local_variable_table_length 1
local_variable_info local_variable_table local_variable_table_length

其中 local_variable_info 项目代表了一个栈帧与源码中的局部变量的关联。

类型 名称 数量
u2 start_pc 1
u4 length 1
u2 name_index 1
u2 descriptor_index 1
u2 index 1

start_pc 和 length 属性分别代表了这个局部变量的生命周期开始的字节码偏移量以及作用范围的长度，两者结合起来就是这个局部变量在字节码之中的作用域范围。
name_index 和 descriptor_index 都是指向常量池中 CONSTANT_Utf8_info 型常量的索引，分别代表了局部变量的名称以及这个局部变量的描述符。
index 是这个局部变量在栈帧局部变量表中 Slot 的位置。在 JDK 1.5 以后引入了泛型而增加增加了一个姐妹属性：LocalVariableTypeTable，以描述泛型擦除的一些信息。

SourceFile 属性

这个属性用于记录生成这个 Class 文件的源码文件名称，也是可选的，主要影响就是抛出异常时能够显示出错代码的文件名，是个定长的属性。

类型 名称 数量
u2 attribute_name_index 1
u4 attribute_length 1
u2 sourcefile_index 1

ConstantValue 属性

ConstantValue 属性的作用是通知虚拟机自动为静态变量赋值，只有被 static 关键字修饰的变量才可以使用这个属性。对于非 static 类型的变量的赋值是在实例构造器方法中进行的，但是对于类变量则有两种方式：在类构造器方法中或者使用 ConstantValue 属性。
目前在 Sun Javac 编译器中，如果同时使用 final 和 static 来修饰一个变量（常量），并且这个变量的类型是基本数据类型或者 String，就会生成 ConstantValue 属性来进行初始化，如果这个变量没有被 final 修饰，或者并非基本类型及字符串，则将会选择在 方法中进行初始化。

类型 名称 数量
u2 attribute_name_index 1
u4 attribute_length 1
u2 constantvalue_index 1

这个是一个定长属性，constantvalue_index 数据项代表了常量池中一个字面常量的引用，根据字段类型的不同，字面量可以是 CONSTANT_Long_info、CONSTANT_Float_info等中的一种。

InnerClasses 属性

这个属性主要用来记录内部类和宿主之间的关联，有内部类的类编译器会为它以及它包含的内部类生成 InnerClass 属性。

类型 名称 数量

u4 attribute_length 1
u2 number_of_classes 1
inner_classes_info inner_classes number_of_classes

u2 inner_class_access_flags 1

其中第一和第二个属性都是指向常量池中 CONSTANT_Class_info 型常量的索引，分别代表了内部类和宿主类的符号引用。
第三个是类名，第四个是访问标识。

Deprecated 及 Synthetic 属性

这两个都是标识类型的布尔属性，只存在有和没有的区别，没有属性值的概念。第一个是表示方法过时了，第二个 Synthetic 属性表示此字段或者方法不是由 Java 源码直接产生的，而是由编译器自行添加的，所有由非用户代码产生的类、方法以及字段都应当至少设置 Synthetic 属性和 ACC_SYNTHETIC 标志位中的一项，唯一的例外就是实例构造器“”和类构造器“”方法。

类型 名称 数量

StackMapTable 属性

这个属性是 JDK1.6 发布后新增到 Class 文件规范中的，它是一个复杂的变长属性，位于 Code 属性表中，这个属性会在虚拟机类加载的字节码验证阶段被新类型检查验证器使用。StackMapTable 属性中包含零至多个栈映射帧（Stack Map Frames），每个栈映射帧都显式或隐式地代表一个字节码偏移量，用于表示该执行到该字节码时局部变量表和操作数栈的验证类型，类型检查验证器会通过检查目标方法的局部变量和操作数栈所需要的类型来确定一段字节码指令是否符合逻辑约束。一个方法的 Code 属性最多只有一个 StackMapTable 属性，否则将抛出 ClassFormatError 异常。

类型 名称 数量
u2 attribute_name_index 1
u4 attribute_length 1
u2 number_of_entries 1
stack_map_frames stack_map_frame_entries number_of_entries

Signature 属性

在 JDK 1.5 以后大幅增强了语法，任何类、接口、初始化方法或者成员的泛型签名如果包含了类型变量（Type Variables）或参数化类型（Parameterized Types），则 Signature 属性会为它记录泛型签名信息，之所以要专门使用这样一个属性去记录泛型类型，是因为 Java 语言的泛型采用的是擦除法实现的伪泛型，在字节码（Code 属性）中，泛型信息编译（类型变量、参数化类型）之后通通被擦除掉。使用擦除法的好处就是实现简单（主要修改 Javac 编译器，虚拟机内部只做了很少的改动）、非常容易实现Backport，运行期也能够节省一些类型所占用的内存空间，但是坏处就是运行期无法将泛型类型与用户定义的普通类型同等对待，例如运行期做反射调用时无法获得泛型信息。Signature 属性就是为了弥补这个缺陷而增设的，现在 Java 的反射 API 能够获取泛型类型，最终的数据来源也就是这个属性。

类型 名称 数量
u2 attribute_name_index 1
u4 attribute_length 1
u2 signature_index 1

其中 signature_index 必须是一个对常量池的有效索引。常量池在该索引处的项必须是 CONSTANT_Utf8_info 结构，表示类签名、方法类型签名或者字段类型签名。

BootStrapMethods 属性

这个属性是在 JDK 1.7 发布后增加到 Class 文件规范中的，它是一个复杂的变长属性，位于类文件属性表中，这个属性主要用于保存 invokedynamic 指令引用的引导方法限定符。如果某个类文件结构的常量池中曾经出现过 CONSTANT_InvokeDynamic_info 类型的常量，那么这个类文件的属性表中必须存在一个明确的 BootstrapMethods 属性，另外，类文件属性表中最多只能有一个 BootStrapMethods 属性。BootStrapMethods 属性与 InvodeDynamic 指令 和 java.lang.Invoke 包关系密切。

类型 名称 数量
u2 attribute_name_index 1
u4 attribute_length 1
u2 num_bootstrap_methods 1
bootstrap_method bootstrap_methods num_bootstrap_methods
类型 名称 数量
u2 bootstrap_method_ref 1
u2 num_bootstrap_arguments 1
u2 bootstrap_arguments num_bootstrap_arguments

字节码指令简介

Java 虚拟机的指令由一个字节长度的、代表某种特定操作含义的数字（操作码，Opcode）以及跟随其后的零至多个代表此操作所需参数（操作数，Operands）而构成的，由于 Java 虚拟机采用面向操作数栈而不是寄存器的架构，所以大多数的指令都不包含操作数，只有一个操作码。
由于限制了 Java 虚拟机中操作码的长度为一个字节（0～255），所以指令集的操作码不可能超过 256 条，又因为 Class 文件格式放弃了编译后代码的操作数长度对齐，这就意味着虚拟机处理那些超过一个字节数据的时候，不得不在运行时从字节中重建出具体的结构，如果要将一个16位的无符号整数用两个无符号字节存储起来的话比如两个字节分别为“b1”和“b1”，那么它们的值如下所示：

字节码与数据类型

在 Java 虚拟机指令集中，大多数的指令都包含其操作所对应的数据类型信息，比如 iload 和 fload，这两条指令的操作在虚拟机内部可能是相同的一段代码来实现的，但是在 Class 文件中它们必须拥有各自的操作码。
由于 Java 虚拟机的操作码的长度只有一个字节，所以面对数量庞大的指令 Java 只提供了有限的类型相关的指令去支持它。有一些单独的指令可以在必要的时候将一些不支持的类型转化为可以支持的类型。大部分的指令都没有支持整数类型 byte、char 和 short，编译器会在编译期或者运行期将它们转为相应的 int 来进行处理的。

加载存储指令

加载和存储指令用于将数据在栈帧中局部变量表和操作数栈中来回进行传输，指令如下所示：

  • 将一个局部变量加载到操作数栈：Tload
  • 将一个数值从操作数栈存储到局部变量表：Tstore
  • 将一个常量加载到操作数栈：Tipush、Tconst
  • 扩充局部变量表的访问索引和指令：wide

运算指令

运算或算术指令用于对两个操作数栈上的值进行某种特定运算，并把结果重新存入到操作栈顶，大体上算术指令分为两种：对整型数据进行运算的指令和对浮点型数据进行运算的指令，对于一些不支持的类型都会转为 int 指令去进行代替。Java 虚拟机仅仅规定在处理数据时只有除法和求余指令在除数出现零的时候才会抛出异常。

  • 加法指令：Tadd
  • 减法指令：Tsub
  • 乘法指令：Tmul
  • 除法指令：Tdiv
  • 求余指令：Trem
  • 取反指令：Tneg
  • 位移指令：Tshl
  • 按位或指令：Tor
  • 按位与指令：Tand
  • 按位异或指令：Txor
  • 局部变量自增指令：Tinc
  • 比较指令：Tcmpg

类型转换指令

Java 支持以下两种类型转换：

  • 宽化类型转换：这是 Java 虚拟机直接支持的，即小范围类型到大范围类型的安全转换，如 int 类型到 long、float、double。
  • 窄化类型转化：这个必须显示的使用转换指令来完成，包括：i2b、i2c、i2s等等，以此类推。

对象的创建与访问指令

虽然类实例和数组都是对象，但是 Java 虚拟机对类实例和数组的创建使用了不同的字节码指令，创建指令如下：

  • 创建类实例的指令：new
  • 创建数组的指令：newarray、anewarray、multianewwarray
  • 访问类字段（static 字段）和实例字段（非 static 字段）的指令：getfield、putfield、getstatic、putstatic
  • 把一个数组指令加载到操作数栈的指令：baload、caload、saload、iaload等等
  • 将一个操作数栈的值存储到数组元素中的指令：bastore、castore 等等
  • 取数组长度的指令：arraylength
  • 检查类实例类型的指令：instanceof、checkcast

操作数栈管理指令

Java 虚拟机提供了一些用于直接操作操作数栈的指令：

  • 将操作数栈的栈顶一个或两个元素出栈：pop、pop2
  • 复制栈顶一个或者两个数值并将复制值或者双份复制值重新压入栈顶：dup、dup2等等
  • 将栈顶的两个数值互换：swap

控制转移指令

控制转移指令可以让 Java 虚拟机有条件无条件的从指定的位置指令继续执行程序，从概念模型上面理解可以认为控制转移指令就是在有条件或无条件的修改 PC 寄存器的值，指令如下：

  • 条件分支：ifeq、iflt、等等
  • 复合条件分支：tableswitch、lookupswitch
  • 无条件分支：goto、jsr、ret等等

方法调用和返回指令

  • invokevirtual：调用对象的实例方法，根据对象的实际类型进行分派。
  • invokeinterface：调用接口方法，它会在运行时搜索一个实现了这个接口方法的对象，找出合适的方法进行调用
  • invokespecial：调用一些需要特殊处理的实例方法，包括实例初始化方法、私有方法和父类方法
  • invokestatic：调用类方法
  • invokedynamic：调用在运行时动态解析出调用点限定符所引用的方法，并执行该方法，前四条调用指令的分派逻辑都固化在 Java 虚拟机内部，而这条的分派逻辑是由用户所设定的引导方法决定的。

异常处理指令

显示抛出的异常由 athrow 指令来实现，还有些会在虚拟机指令检测到异常时自动抛出，在 Java 虚拟机中，处理异常（catch 语句）不是由字节码指令来实现的，而是用异常表来完成的。

同步指令

Java 虚拟机支持方法级的同步和方法内部的同步，两种同步结构都是使用管程（Monitor）来支持的。方法级的同步是隐式的，即无需通过字节码指令来控制，它实现在方法调用和返回操作中。虚拟机可以从方法常量池的方法表结构中的 ACC_SYNCHRONIZED 访问标志得知一个方法是否为同步方法，如果是的话那么在调用这个方法的时候就会先去持有当前同步锁的管程，然后执行方法，无论方法执行失败与否，最后都会释放管程。
Java 虚拟机中有 monitorenter 和monitorexit 两条指令来支持 synchronized 关键字的语义，通过分析字节码序列我们可以直到为了保证在方法执行异常的时候 monitorenter 和 moniterexit 指令依旧可以正确的配对执行，编译器会自动产生一个异常处理器。用来执行 monitorexit 指令。

总结

虽然 Java 技术体系发生了很大的变化，但是 Class 文件结构一致都比较稳定，Class 文件是 Java 虚拟机执行引擎的数据入口，了解 Class 文件的结构对后面了解虚拟机的执行引擎很重要，所以这一篇中我尽可能的将 Class 的每部分的定义、数据结构和使用方法记录下来。
