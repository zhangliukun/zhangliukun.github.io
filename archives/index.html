深入理解Java虚拟机 读书笔记（五）虚拟机类加载机制

3月 12, 2017 发布在 java

概述

虚拟机把描述类的数据从 Class 文件加载到内存，并对数据进行校验、转换解析和初始化，最终形成可以被虚拟机直接使用的 Java 类型，这就是虚拟机的类加载机制。
与那些编译时需要进行链接的语言不同，Java 里面类型的加载、链接和初始化过程都是在程序运行期间完成的，Java 的动态扩展语言特性就是依赖运行期动态加载和动态链接这个特点实现的。用户可以通过 Java 预定义的和自定义的类加载器来使用运行时从网络或者其他地方加载的一个二进制流作为程序代码的一部分。

类加载的时机

类从被加载到虚拟机内存中开始，到卸载出内存为止，整个的生命周期包括：

  • 加载（Loading）
  • 验证（Verification）
  • 准备（Preparation）
  • 解析（Resolution）
  • 初始化（Initialization）
  • 使用（Using）
  • 卸载（Unloading）

概述

如今编写成二进制本地机器码已经不是唯一的选择了，越来越多的程序语言选择了与操作系统和机器指令集无关的、平台中立的格式作为程序编译以后的存储格式。各种不同的平台的虚拟机与所有的平台统一使用的程序存储格式是“字节码（ByteCode）”，实现语言无关性的基础仍然是虚拟机和字节码存储格式。Java 虚拟机不和任何语言绑定，它只和“Class 文件”这种特定的二进制文件格式所关联，Class 文件中包含了 Java 虚拟机指令集和符号表以及若干其他的辅助信息。主要能编译成 Class 文件，Java 虚拟机并不关心来源是何种语言，所以Glojure、Groovy、Scala、JRuby、Jython 这些能够编译成 Class 文件的语言都可以在 Java 虚拟机上面运行。

深入理解Java虚拟机 读书笔记（三）垃圾收集器和内存分配策略

2月 24, 2017 发布在 java

垃圾收集器关心的内存

  • 程序计数器、虚拟机栈、本地方法栈这三个区域是线程私有的，生命周期和线程相同。
  • 栈中的栈帧随着方法的进入和退出规则的入栈和出栈，每一个栈帧中分配的内存在类结构确定后基本就是已知的。

这几个区域的内存分配都是确定的，不需要过多考虑回收的问题，因为方法结束或者线程结束以后，内存自然就回收了。

  • Java 堆和方法区则不一样，一个接口中的多个实现类所需要的内存也不一样，我们只有在程序出于运行中才能知道会创建哪些对象，这部分的内存的分配和回收都是动态的，垃圾回收主要关注的是这部分内存。

前言

一些框架的痛点

最近准备打造一款 material design 风格的 github 的 Android 客户端，在实现搜索仓库列表界面时，涉及到列表肯定要下拉刷新，上拉加载更多，以前项目中用到的都是一个开源项目 Android-PullToRefresh，但是这个仓库介绍中发现以及标记为 DEPRECATED 了 ，这个框架对于一般的需求确实够用了，但是不支持 RecyclerView ,这个开源框架的痛点就是如果要支持某一个控件的下拉刷新功能，就需要对这个控件进行适配修改，在控件这么多的时代，如果要全部支持那是一个巨大工作量，因为这个框架不是通用性的，估计这也是这个框架的开发者将之废弃的原因吧。

我所想要的

  • 支持下拉刷新，上拉加载并且对不同的控件通用低耦合
  • 支持多种不通的数据类型显示
  • 支持多种个性化 Header 和 Footer
  • 易用性高

Android内存管理

7月 20, 2016 发布在 android

管理你的app内存

在所有的软件开发环境中内存是一个非常珍贵的资源，但是在物理内存被限制的移动操作系统中内存更加珍贵，虽然Android的Dalvik虚拟机会执行常规的垃圾收集操作，但是你不应该忽视你的app的内存的在哪里得到分配和释放。

为了垃圾收集器从你的app中回收内存，你需要避免引入内存泄漏（通常是由于在全局的成员里面保持了堆对象的引用）并且在恰当的时候释放掉被引用的对象，对于大多数的app，Dalvik的垃圾收集器会帮你处理剩下的事：系统会回收你的内存分配当符合条件的对象不在你的app的正在运行的线程中被引用到。

这篇文章解释了Android如何管理app的进程和内存分配的，并且介绍了你在Android开发时怎么样才能主动减少内存的使用，如果要获取更多的关于在进行java编程时清理你的资源的标准实践你可以参考其他关于管理资源引用的书或者在线文档。如果你正在寻找关于分析你的app的内存使用情况的信息的话，可以读Inverstingating Your RAM Usage
