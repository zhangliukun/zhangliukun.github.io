管理你的app内存

在所有的软件开发环境中内存是一个非常珍贵的资源，但是在物理内存被限制的移动操作系统中内存更加珍贵，虽然Android的Dalvik虚拟机会执行常规的垃圾收集操作，但是你不应该忽视你的app的内存的在哪里得到分配和释放。

为了垃圾收集器从你的app中回收内存，你需要避免引入内存泄漏（通常是由于在全局的成员里面保持了堆对象的引用）并且在恰当的时候释放掉被引用的对象，对于大多数的app，Dalvik的垃圾收集器会帮你处理剩下的事：系统会回收你的内存分配当符合条件的对象不在你的app的正在运行的线程中被引用到。

这篇文章解释了Android如何管理app的进程和内存分配的，并且介绍了你在Android开发时怎么样才能主动减少内存的使用，如果要获取更多的关于在进行java编程时清理你的资源的标准实践你可以参考其他关于管理资源引用的书或者在线文档。如果你正在寻找关于分析你的app的内存使用情况的信息的话，可以读Inverstingating Your RAM Usage

Android如何管理内存

Android并不提供内存的交换空间，但是它确实使用了分页和内存映射(mmapping)来管理内存，这意味着任何你修改的内存－无论是分配新的对象或者使用了内存映射的页－会在RAM中保持常驻并且不能被置换出页，所以唯一的将内存从你的app中释放的方法就是释放你可能对对象保持的引用，使得垃圾收集器可以收集到这些内存。这里有一个例外：任何没有被更改过的内存映射的文件，比如代码，如果系统想要使用这部分内存的话这部分文件能够被分页置换出去。

共享内存

为了能够满足在内存中的一切需要，Android尝试跨进程共享RAM页，可以通过以下几种方法来实现：

  • 每个app的进程是从一个已经存在的叫Zygote的进程中被fork的，这个Zygote进程会在系统启动和载入一些通用的系统框架代码和资源（比如activity主题）的时候被启动。为了启动一个新的app的进程，系统会fork这个Zygote进程然后将app的代码载入运行在新的进程中，这允许为系统框架代码和资源分配的大部分的内存页能够在所有的app进程中共享。
  • 大多数的静态数据会被内存映射到一个进程中。这不仅允许相同的数据能够在进程中被共享，也能允许当需要的时候它能够被置换出去。示例静态数据包括：Dalvik代码（通过将它放在一种预链接的.odex文件中来进行直接内存映射），并且将传统项目的元素像本地代码放在.so文件中。
  • 在很多地方，Android通过使用明确分配的共享内存区(无论是ashmem或者gralloc)跨进程共享相同的动态内存。例如，window surfaces use shared memory between the app and screen compositor, and cursor buffers use shared memory between the content provider and client。

由于共享内存的大范围使用，决定你的app使用多少内存是非常值得关注的。关于合理决定你的app的内存使用情况的技术讨论可在这里看到Investigating Your RAM Usage

分配和回收app内存

这里有一些实例关于Android是如何从你的app中分配和回收内存的：

  • Dalvik中每一个进程的堆大小都被限制在了一个单独的虚拟内存范围内。这定义了逻辑堆的大小，这个大小可以随着需要而增长（但是最多只能达到系统分给每个app的大小）。
  • 堆的逻辑大小和堆使用的物理空间的大小是不同的。当检查你的app的堆时，Android计算出了一个叫做Proportional Set Size (PSS)，这个比例设置大小占据了呵其他进程共享的dirty和clean的pages－但是只有在一个由多少app共享这个内存的比例的量。这个（PSS）的总量是系统认为你的无礼内存占用。
  • 这个Dalvik堆不会压缩你的堆的逻辑大小，这意味着Android不会整理堆到关闭空间。Android只会在堆尾部没有被使用的空间的时候才会减少逻辑堆的大小，但这并不意味着被堆使用的物理内存不能缩小。在垃圾收集以后，Dalvik会遍历堆并且寻找未被使用的页，然后使用madvise来返回这些页到内核。所以，对大的块的成对的分配和释放会导致回收所有的（或者几乎所有的）所使用的物理内存。但是，从小的分配的块中回收内存可能会变得十分低效，因为这个被小块使用的页可能还与其他没有被释放的东西所共享。

限制App内存

为了维持一个可运行的多任务的环境，Android对每个app设置了一个苛刻的堆大小限制，精确的堆大小限制的值在不同的设备之间是不同的，这取决于这个设备总共有多少内存可以使用，如果你的app已经达到了堆限制的大小并且继续尝试申请分配更多的内存，那么就会OOM了。

有时候，你想要查一下你能够在这个设备上面获取到多少内存堆空间－比如，决定多大的数据保存到缓存中是安全的，你可以调用getMemoryClass()，这会返回一个你的app能够拥有的堆内存的以M为单位的整数值。更多讨论参考Check how much memory you should use

App之间的切换

当用户在app之间进行切换的时候，Android会保持那些不在用户可视范围的app组件的进程在一个LRU的缓存中，例如，当用户第一次启动app时，会启动一个进程，但是当用户离开这个app的时候，这个进程不会马上退出，系统会将它保存到一个缓存中，所以当用户后来返回到这个app的时候，这个进程能够在app切换的时候很快被重用。

如果你的app有一个已经被缓存的进程并且这个进程保持了现在不需要使用的内存资源，那么你的app－即使你不在使用它－这还是会限制系统的总体性能表现。所以，当系统在低内存下运行时，它可能会杀掉在LRU缓存中的最近最少使用的进程，但是也会考虑到哪些进程是内存密集的。为了保持你的进程能够尽可能多的被缓存，遵循以下的关于何时区释放你的引用的建议。

谨慎的使用services

如果你使用一个service来做一些事情，不要一直让它运行除非它确实在运行某些任务，另外你也要特别的小心不要因为这个service停止失败而泄露内存当它的工作做完的时候。

当你开始启动一个service的时候，在这个service运行的时候系统总会保留这个进程，这使得整个进程非常昂贵因为被这个service使用的RAM不能够被其它任何东西使用或者将这个RAM分页置换出去page out，这样的话使得系统减少了能够在LRU cache中保持的进程的数量，使得app的切换低效。这甚至会导致系统在内存紧张的时候发生系统颠簸并且这个系统不能够保持足够的进程来支撑所有的在运行的service

最好的来限制你的service的寿命的方法就是使用一个IntentService,这个会在完成移动它的Intent的任务以后结束自己。更多信息可以阅读Running in a Background Service

当一个service不再被需要的时候还让它运行是android app中最严重的内存管理错误之一，所以在保持的你的app的service运行的时候不要太贪婪，这不仅会犹豫RAM的限制提高你的app表现差劲的风险，还会使得用户发觉你的app太差了从而卸载app

当你的用户界面被隐藏时释放内存

当用户切换到一个不同的app并且你的UI不再可见的时候，你应该释放掉只被你使用的那些UI资源，在这个时候释放掉你的UI资源可以显著的提高系统的缓存进程的容量，对用户体验来说又一个直接的影响。

为了能够在用户退出你的UI的时候收到通知，可以实现onTrimMemory()方法在你的Activity类的callback中，你应该用这个方法来监听TRIM_MEMORY_UI_HIDDEN等级，这个会通知你的UI现在被隐藏了你应该释放只被你使用的UI资源了。

注意当你的所有的app的进程的UI组件对用户不可见的时候你的app接收到有TRIM_MEMORY_UI_HIDDEN等级的onTrimMemory()的回调，这个和onStop()的callback不同，因为onStop()在你的app内的activity切换时也会触发，这时候很多情况下你不应该释放掉你的UI资源除非你收到onTrimMemory(TRIM_MEMORY_UI_HIDDEN)。这样才能保证用户从其它app返回到你的app的时候，你的UI资源仍然存在并且能够快速恢复你的activity。

当内存紧张时释放内存

在你的app的生命周期内，onTrimMemory()这个callback会告诉你什么时候整体的设备内存很少了，你应该基于以下的由onTrimMemory()传递出来的内存等级来进一步的释放资源来做出回应。

  • TRIM_MEMORY_RUNNING_MODERATE
  • TRIM_MEMORY_RUNNING_LOW
  • TRIM_MEMORY_RUNNING_CRITICAL
  • TRIM_MEMORY_BACKGROUND
  • TRIM_MEMORY_MODERATE
  • TRIM_MEMORY_COMPLETE

具体相关解释可以查看原文。

**Note**:当系统开始清理LRU缓存中的进程的时候，尽管它主要的自下而上的工作的，它确实在哪个进程消耗了更多的内存和哪个进程被杀死以后能够获得更多的内存上面作出了一些考虑，所以你在LRU的list中的整体消耗的内存越少，你能够留在list中的机会就越大从而能够更快的恢复。

检查你应该使用多少内存

像早先提到的那样，每个android设备都有不同的RAM大小给系统使用，并且以此来给每个app提供一个不同的堆大小限制，你可以调用getMemoryClass()来得到一个以M为单位你的app可以使用的堆大小的估值，如果你的app尝试分配比它所能够使用的多的内存的时候，它会收到一个OutOfMemoryError。

在十分特殊的场景下，你可以申请更大的堆空间通过将在manifest中的tag的largeHeap属性设置为true，如果你这样做的话，你可以调用getLargeMemoryClass()来获取到一个更大的堆内存空间的估值。

但是，这个获取更多的堆空间的方法只是为一小部分的确实需要消费更多的RAM的app设计的（比如一个大型的照片编辑app），永远不要简单的因为你超出了内存而申请更大的堆空间并且你需要一个快速修复的方法－你应该只有在你精确的知道你的内存是在哪里被分配的并且为什么必须被保留。另外，及时你很确信你的app有正当理由获取大内存，你应该避免在任何时候都去申请它。使用多余的内存将会提高对用户的整体使用体验因为垃圾收集将会花更多的时间并且系统表现会变得更慢当发生任务切换或者执行其它的操作的时候。

另外，大的堆空间大小在不同的设备上时不同的，当在一些内存有限的设备上运行的时候，大的堆大小可能与常规的堆大小完全相同，所以即使你确实申请了大的堆大小，，你应该通过调用getMemoryClass()来检查常规堆的大小并且确保总是低于那个限制。

避免在bitmaps上浪费内存

当你载入一张图片时，只有当图片的分辨率是你所需要的设备屏幕的分辨率时才将它保存在内存中，如果原始图片的分辨率更高的话将分辨率降低，记住图片的分辨率的增加(increase)会导致相应的内存的(increase^2)的增加，因为X和Y的维度都在增加。

**Note**:在Android 2.3x(API level 10)以前，图片对象总是在app的堆内存上面显示出相同的大小而不管图片分辨率的大小（真正的数据是单独存储在本地内存中的），这使得我们很难进行图片内存分配的调试因为我么的堆内存分析工具不能看到本地的内存分配，但是，从Android3.0(API level 11)开始，图像数据的分配会在你的app的Dalvik的堆上，改善的垃圾回收和调试。所以如果你使用图片时并且你想了解为什么你的app在旧的设备上面使用了一些内存的话，可以切换到一个Android 3.0或者更高的系统上面去调适

更多的关于图片的建议，可读Managing Bitmap Memory

使用优化过的数据容器

发挥Android框架中优化过的数据容器的优势，比如SparseArray,SparseBooleanArray,LongSparseArray，通用的HashMap实现是十分低效的因为每个映射需要一个单独的入口对象。另外，SparseArray类是更加的高效的因为它能避免key的自动装箱和临时值（创建另一个对象或者两个条目），并且不需要害怕降为原始的数组当需要的时候。

要注意内存开销

要清楚你使用的语言和类库的开销，并且当你设计你的app时时刻记在脑中，从开始到结束，表面上看上去无害的事物实际上可能会造成非常大的开销，如下例子：

  • Enums枚举经常需要比静态常量两倍多的内存。你应该避免在Android中使用枚举类型。
  • 每个在java中的类（包括匿名内部类）使用了约500字节的代码。
  • 每个类实例有12－16字节的内存开销。
  • 将一个单独的条目放入一个HashMap中需要额外的为类条目分配大约32字节（可以参考之前的优化的数据容器）

这里的和那里的小部分字节总和增加很快，app设计中类或者对象很多的将会遭受这部分开销的苦难，这个会使你很难进行堆内存分析并且很难意识到你的问题是很多小的对象在使用你的内存。

要小心代码抽象

平常开发者都会使用抽象作为“良好的编程实践”，因为抽象能够提高代码的自由和维护性，但是，抽象也会带来更大的开销：通常它们需要一个合理的数量的代码来扩展它们，需要更多的时间和内存来为那些代码在内存中映射，所以如果你的抽象不能带来很好的收益，你应该避免它们。

使用 nano protobufs 来序列化数据

Protocol buffers是一个由谷歌为序列化结构数据而设计的语言中性、平台无关、可扩展的机制－想想XML，但是更小，更快，更简单。如果你决定为你的数据使用protobufs，你应该总是在你的客户端代码中使用nano prorobufs,普通的protobufs整合了相当多的冗余的代码，这将会引起很多的问题：内存使用率增高，签名后的APK大小增大，执行速度变慢，更快的达到DEX方法数极限。

更多的关于“nano version”的章节可看 protobuf readme

避免使用依赖注入框架

使用一个像Guice或者RoboGuice这样的依赖注入框架可能比较受欢迎因为它能简化你的代码并且提供了一个对于测试和配置改变都很有用的可调整的环境，但是，这些框架倾向于执行很多的进程初始化通过扫描那些你用注解标注的代码，这会要求将很多的即使你并不需要使用的代码也映射到内存中，这些映射后的内存页是被分配到干净的内存中所依Android可以将它们丢弃，但是这只能在这些页已经离开内存很长一段时间以后才能发生。

谨慎使用第三方类库

第三方的类库经常并不是为了移动端而进行设计的，当用在手机客户端时可能非常低效。至少，当你决定使用第三方类库的时候，你应该知道你会在对第三方库进行移植和维护的时候承受巨大的负担来对移动端进行优化。提前做计划并且在你决定使用它之前根据类库的代码大小和内存使用情况来分析它。

即使这个类库时为移动端设计的它也是有潜在的危险的，因为每个类库做的事情都不一样，例如，一个类库可能使用的是nano的protobufs，但是另外的一个可能使用的是micro的protobufs，现在你有两个不同的protobuf的实现。这也会发生在对于logging，analytics，image loading，cache和其他所有的你想不到的不同的实现中，ProGuard并不会拯救你因为你想要用这个类库所实现的功能都是低等级的依赖。这会变得尤其麻烦当你从一个类库中来使用一个Activity的子类（它往往会有大片的依赖），当类库使用反射的话（这很普遍也意味着你需要花大量的时间来手动调整ProGuard来使它能够工作），诸如此类的。

你也需要注意不要掉进一个为了某个共享库的几十个特性中的一个或者两个特性而使用它的陷阱，你肯定不想拉入大量的你不需要使用的代码和增大开销，最后，如果没有一个非常适合你所需要的已有的实现，可能最好的方法就是你自己去实现它。

使用ProGuard来去除不需要的代码

ProGuard工具能够除去，优化和混淆你的代码，它能够移除不使用的代码和通过一些语意上比较难理解的名字来重新命名你的类、成员和方法，使用ProGuard能够压缩你的代码，能够减少你的代码映射入内存中的页。

在你的最终的APK上面使用zipalign

If you do any post-processing of an APK generated by a build system (including signing it with your final production certificate), then you must run zipalign on it to have it re-aligned. Failing to do so can cause your app to require significantly more RAM, because things like resources can no longer be mmapped from the APK.
如果你对一个有生成系统生成的APK做任何的后处理（包括使用你的最终的产品证书对APK进行签名），你必须运行zipalign对它进行重新排列，如果你不这样做的话会使你的app显著的提高内存的使用量，因为像一些资源文件将不再会从APK中进行内存映射。

Note: Google Play Store不会接受那些没有进行zipalign的APK

分析你的内存使用

一旦你达到了相对稳定的构建，你就应该开始分析你的app在整个生命周期中使用了多少内存。更多关于分析app使用参考Investigating Your RAM Usage

使用多个进程

如果合适的话，有一项比较先进的管理你的app内存的技术就是将你的app划分成几个组件放进不同的进程中，这个技术必须要小心的使用因为大多数的app不会运行在不同的进程中，因为这个会很容易的增加而不是减少你的内存使用量如果你使用不当的话。这项技术主要对那些需要在后台运行和前台一样十分重要的任务并且能够独立的运行的时候十分有效。

一个比较适合使用多进程的例子就是当你构建一个音乐播放器的时候并且它需要长时间的在后台通过一个service来播放音乐。如果整个app运行在一个进程中，那么当它播放音乐的时候，许多为它的activity的UI所分配的资源也必须被保存在内存中，即使用户现在已经完全在其他的app中，这个service还在进行播放。像这种app就应该被划分为两个进程：一个给UI，一个给那个不断在后台进行任务的service。

另外，当运行多于一个进程的时候，尽可能的保持你的代码高效是十分重要的，因为任何通常实现的不必要的内存开销现在会复制进每个进程中。例如，如果你使用枚举（you should not use enums),所有的内存所需要的将会在每个进程中创建和初始化这些常量，并且所有的你使用的适配器抽象和临时变量或者其他的开销都会相似的被复制。

另一个对多进程需要关心的是存在它们之间的依赖。例如，如果你的app有一个持有你的UI的运行在默认进程中的content provider，那么使用那个content provider的后台进程也会要求你的UI进程保存在内存中，如果你的目标是创建一个能够独立于重UI进程的后台进程的话，你不能对那些在UI进程中执行的content provider或者services有依赖。
